# Pokerface 2.0 命令行联机 + 自定义人机 PRD

## 1. 背景

Pokerface 1.0 已经具备完整德州扑克核心能力：2-9 人牌桌、标准发牌与下注流程、牌型判定、边池结算、水下模式、AI 难度和命令行交互。

2.0 的目标不是做网页或移动端，而是在现有核心逻辑基础上增加命令行联机能力。联机服务端部署在个人服务器上，多个朋友可以在各自终端里连接服务器、加入同一房间，和真人玩家、自定义 AI 一起打牌。

## 2. 产品定位

做一个轻量、朋友局优先的命令行联机德州扑克小游戏。

核心原则：

- 继续以终端文字交互为主。
- 支持个人服务器常驻开服，多人通过命令行加入。
- 支持真人玩家和 AI 玩家混合牌桌。
- 房主可添加、移除、命名和配置 AI。
- 服务端统一发牌、校验动作、推进牌局和结算。
- 不做网页 UI、移动端、公开大厅、账号系统或真钱玩法。

## 3. 用户目标

### 3.1 房主

- 连接个人服务器并创建一个牌局房间。
- 把连接地址、端口和房间码发给朋友。
- 在命令行等待区管理座位和 AI。
- 开始游戏、继续下一手、结束房间。

### 3.2 普通玩家

- 通过命令行连接朋友的房间。
- 输入昵称并入座。
- 在终端里看到公共牌、自己的手牌、座位状态、下注状态和最近行动。
- 轮到自己时输入命令行动。
- 断线后可在房间生命周期内重连回原座位。

## 4. 范围

### 4.1 2.0 必须支持

- 在个人服务器上启动命令行服务端。
- 命令行客户端加入服务端。
- 私密房间码。
- 2-9 个座位。
- 每个座位可以是真人、AI 或空位。
- 至少 2 个可参与座位才能开局。
- 可由 1 名真人 + 多个 AI 开局。
- 多名真人玩家同时在线参与同一牌局。
- 房主在开局前配置：
  - 总座位数。
  - 初始筹码。
  - 小盲注和大盲注。
  - 水下模式。
  - 行动超时时间，超时默认弃牌。
  - AI 默认配置。
- 房主在等待区管理 AI：
  - 添加 AI。
  - 移除 AI。
  - 修改 AI 名称。
  - 修改 AI 难度。
  - 修改 AI 内部策略。
- 服务端只向玩家发送其可见信息。
- 玩家断线后可重连。
- 单手结束后可继续下一手。

### 4.2 2.0 暂不支持

- 图形界面。
- 浏览器版联机。
- 移动端适配。
- 公开大厅和随机匹配。
- 账号、密码、好友系统。
- 数据库持久化。
- 长期战绩、排行榜。
- 金钱、充值、提现。
- 多桌锦标赛。

## 5. 运行方式

### 5.1 部署原则

2.0 首选部署方式：

- 服务端运行在个人服务器上。
- 公司电脑只用于开发、提交代码或远程触发部署。
- 公司电脑不开放公网端口。
- 公司电脑不使用 ngrok、Tailscale 之类公网隧道作为正式联机方案。
- 朋友只连接个人服务器地址和房间码。

这样可以避免把公司电脑暴露给外部连接，也更方便做稳定联机测试。

### 5.2 启动服务端

示例：

```bash
node poker.js server --host 0.0.0.0 --port 3000
```

或后续拆分为：

```bash
node server.js --host 0.0.0.0 --port 3000
```

启动后服务端依次询问或接受参数：

- 监听地址，默认 `0.0.0.0`。
- 监听端口，默认 `3000`。
- 管理口令，默认启动时随机生成。
- 是否允许创建多个房间，P0 默认否。

服务端启动后输出：

```text
Pokerface 联机服务已启动
地址：0.0.0.0
端口：3000
管理口令：M7KQ-29FD

房主创建房间命令：
node poker.js create your-server.com:3000
```

### 5.3 创建房间

房主在自己的终端运行：

```bash
node poker.js create your-server.com:3000
```

客户端连接后：

1. 输入昵称。
2. 输入服务器管理口令，或使用服务端允许的房主 token。
3. 设置房间参数。
4. 创建房间并获得房间码。
5. 进入等待区。

创建成功后输出：

```text
房间已创建
服务器：your-server.com:3000
房间码：739251

朋友加入命令：
node poker.js join your-server.com:3000 739251
```

### 5.4 加入房间

示例：

```bash
node poker.js join your-server.com:3000 739251
```

客户端连接后：

1. 输入昵称。
2. 选择空座位，或输入 `旁观` 等待。
3. 进入等待区。
4. 等房主开始游戏。

### 5.5 本机开发调试

为了方便开发调试，支持一条命令同时启动本地服务端并创建本地客户端：

```bash
node poker.js host
```

该模式等价于：

- 启动本地服务端。
- 当前终端作为房主客户端加入。
- 仅用于本机或内网测试，不作为跨网络朋友联机的推荐方式。

## 6. 核心用户流程

### 6.1 创建房间

1. 服务端已在个人服务器上运行。
2. 房主运行 `create` 命令连接服务器。
3. 系统创建房间并生成房间码。
4. 房主进入命令行等待区。
5. 房主把加入命令发给朋友。

### 6.2 玩家加入

1. 玩家运行 `join` 命令连接个人服务器。
2. 输入昵称。
3. 客户端展示座位表。
4. 玩家输入座位号入座。
5. 等待房主开始。

### 6.3 添加自定义 AI

房主在等待区输入：

```text
添加AI
```

或快捷命令：

```text
bot add
```

然后依次配置：

- 座位号。
- AI 名称。
- 难度：简单 / 普通 / 困难。
- 内部策略模板，默认不向其他玩家展示。

### 6.4 开始游戏

房主在等待区输入：

```text
开始
```

服务端校验：

- 至少 2 个可参与座位。
- 所有真人座位均在线，或房主确认允许离线座位跳过。
- 盲注和筹码配置合法。

校验通过后，服务端洗牌、发牌、扣盲注并进入第一手牌。

### 6.5 玩家行动

1. 服务端向所有客户端广播公共状态。
2. 服务端只向当前玩家展示可输入动作。
3. 当前玩家输入命令。
4. 客户端提交动作到服务端。
5. 服务端校验动作。
6. 服务端应用动作并广播结果。

### 6.6 断线重连

1. 玩家断线后，座位标记为「离线」。
2. 服务端保留座位和筹码。
3. 玩家用同一连接命令重新加入。
4. 输入昵称后，若本地保存有会话标识，自动恢复座位。
5. 如果无法自动恢复，允许输入重连码恢复座位。
6. 当前行动玩家离线或未输入超过动作倒计时，服务端自动执行弃牌。

## 7. 命令行交互

### 7.1 等待区展示

等待区至少展示：

```text
房间 739251 | 等待中
配置：6 人桌 | 初始筹码 1000 | 盲注 5/10 | 水下：开 | 超时：60 秒

座位：
1. 你（房主）      真人  在线
2. Alice          真人  在线
3. 空
4. AI-1           AI    在线
5. 空
6. 空

房主命令：
开始/start，添加AI/bot add，移除AI/bot remove，设置/config，踢出/kick，退出/q

玩家命令：
入座 位置/sit 位置，离座/leave，刷新/s，退出/q
>
```

### 7.2 牌局展示

沿用 1.0 的文字牌桌格式，并增加联网信息：

- 房间码。
- 当前在线 / 离线状态。
- 当前行动玩家。
- 自己是否是当前行动玩家。
- 动作倒计时，倒计时结束后默认弃牌。

示例：

```text
房间 739251 | 第 8 手 | 翻牌圈 | 轮到 Alice | 剩余 42 秒
公共牌：A红桃 9方片 3梅花
你的手牌：K黑桃 K红桃

底池：180
当前最高下注：80
你需要跟注：40

座位：
庄  你       在线  筹码 920   本轮 40   等待
小  Alice    在线  筹码 760   本轮 80   行动中
大  AI-1     AI    筹码 1180  本轮 80
    Bob      离线  筹码 600   本轮 0    已弃牌

最近行动：
AI-1 加注到 80
你 跟注 40

等待 Alice 行动...
```

轮到自己时：

```text
请输入：弃牌/f，跟注/c，加注 金额/r 金额，全下/a，状态/s，退出/q
>
```

## 8. 玩家命令

### 8.1 等待区命令

| 中文命令 | 快捷命令 | 含义 |
| --- | --- | --- |
| `入座 3` | `sit 3` | 坐到 3 号座位 |
| `离座` | `leave` | 离开当前座位 |
| `准备` | `ready` | 标记自己准备好，P1 |
| `状态` | `s` | 刷新房间状态 |
| `退出` | `q` | 离开房间 |

### 8.2 房主命令

| 中文命令 | 快捷命令 | 含义 |
| --- | --- | --- |
| `开始` | `start` | 开始第一手或下一手 |
| `添加AI` | `bot add` | 添加 AI |
| `移除AI 4` | `bot remove 4` | 移除 4 号座位 AI |
| `设置AI 4` | `bot config 4` | 修改 4 号座位 AI |
| `设置` | `config` | 修改房间配置 |
| `踢出 2` | `kick 2` | 移除 2 号真人座位 |
| `关闭房间` | `close` | 关闭房间 |

### 8.3 牌局行动命令

沿用 1.0：

| 中文命令 | 快捷命令 | 含义 |
| --- | --- | --- |
| `弃牌` | `f` | 放弃本手牌 |
| `过牌` | `c` | 无需跟注时让过 |
| `跟注` | `c` | 有下注时补齐到当前最高下注 |
| `下注 50` | `b 50` | 下注 50 |
| `加注 120` | `r 120` | 加注到 120 |
| `全下` | `a` | 全下 |
| `状态` | `s` | 重新显示当前牌桌状态 |
| `退出` | `q` | 断开连接或退出游戏 |

## 9. 自定义 AI

### 9.1 基础配置

每个 AI 支持：

- 名称：默认 `AI-1`、`AI-2`。
- 难度：简单 / 普通 / 困难。
- 内部策略模板：
  - 稳健：偏保守，少 bluff。
  - 激进：更常下注和加注。
  - 松凶：入池多，进攻多。
  - 紧弱：入池少，遇压力更容易弃牌。
  - 娱乐：随机性更强。
- AI 的内部策略、bluff 频率、进攻倾向等配置只用于服务端决策，不在等待区、牌局座位表或行动日志中展示。公共界面只展示 AI 名称和 AI 身份。

### 9.2 行为约束

- AI 决策只在服务端执行。
- AI 只能读取自己的手牌、公共牌和公共行动历史。
- AI 不能读取牌堆未来牌。
- AI 不能读取真人或其他 AI 的暗牌。
- AI 生成的动作必须经过同一套动作校验。

### 9.3 高级参数，P1

- 入池倾向：tight / loose。
- 进攻倾向：passive / aggressive。
- bluff 频率：低 / 中 / 高。
- 跟注韧性：低 / 中 / 高。
- 加注尺度：小 / 标准 / 大。
- 行动延迟：快速 / 正常 / 慢速。

## 10. 牌局规则

2.0 沿用 1.0 的核心规则：

- 标准 52 张牌。
- 德州扑克 2 张手牌 + 5 张公共牌。
- 翻牌前、翻牌、转牌、河牌四轮下注。
- 牌型判定从皇家同花顺到高牌。
- 支持弃牌、过牌、跟注、下注、加注、全下。
- 支持边池结算。
- 支持水下模式。

2.0 额外规则：

- 服务端是唯一权威状态。
- 客户端只提交命令，不直接修改牌局。
- 每个玩家只能收到自己的暗牌。
- 未摊牌玩家的暗牌不能被其他客户端看到。
- 本手提前结束时，不公开弃牌玩家暗牌。
- 所有玩家看到一致的公共牌、底池、下注和行动日志。

## 11. 技术方案

### 11.1 总体原则

- 复用现有 `GameEngine`、`Deck`、`HandEvaluator`、`BotPlayer`。
- 将核心逻辑从「本地输入输出驱动」改为「服务端事件驱动」。
- 命令行客户端只负责展示、输入和网络通信。
- 服务端维护房间、座位、连接、权限和完整牌局状态。

### 11.2 建议进程模型

支持四种运行形态：

- `server`：在个人服务器上启动常驻服务，不占座。
- `create`：连接服务器并创建房间，当前终端成为房主客户端。
- `join`：启动客户端并连接远端服务端。
- `host`：本机开发调试模式，启动本地服务端并自动创建房主客户端。

### 11.3 通信协议

推荐优先使用 Node.js 内置 `net` TCP socket，避免引入复杂依赖。

消息格式使用一行一个 JSON：

```json
{"type":"player_action","roomCode":"739251","seatId":"seat-1","action":{"kind":"call"}}
```

选择 JSON Lines 的原因：

- 终端联机足够简单。
- 便于调试和日志记录。
- Node 内置能力即可实现。
- 后续可平滑替换为 WebSocket 或 HTTP。

### 11.4 客户端发往服务端

- `hello`
- `join_room`
- `reconnect`
- `sit_down`
- `leave_seat`
- `add_bot`
- `update_bot`
- `remove_bot`
- `update_config`
- `start_game`
- `player_action`
- `next_hand`
- `close_room`
- `ping`

### 11.5 服务端发往客户端

- `welcome`
- `room_snapshot`
- `game_snapshot`
- `private_state`
- `legal_actions`
- `action_error`
- `player_connected`
- `player_disconnected`
- `hand_finished`
- `room_closed`
- `pong`

### 11.6 引擎改造要求

现有 `GameEngine` 中输入输出耦合较强，2.0 需要拆成可被服务端调用的无 UI 核心：

- `startHand()`：开始一手牌。
- `getVisibleState(seatId)`：返回某个座位可见状态。
- `getPublicState()`：返回公共状态。
- `getLegalActions(seatId)`：返回当前座位可执行动作。
- `applyAction(seatId, action)`：校验并应用动作。
- `advance()`：推进阶段、触发 AI 行动、结算。
- `serialize()` / `deserialize()`：支持测试和房间快照。

### 11.7 建议模块

- `core/`
  - `GameEngine`
  - `Deck`
  - `HandEvaluator`
  - `PotManager`
  - `BotPlayer`
  - `ActionValidator`
- `server/`
  - `Room`
  - `RoomManager`
  - `SeatManager`
  - `TcpGateway`
  - `SessionManager`
- `client/`
  - `CliClient`
  - `CliRenderer`
  - `CommandParser`
  - `ReconnectStore`

## 12. 服务器部署需求

### 12.1 服务器环境

P0 服务器要求：

- Linux 服务器。
- Node.js LTS。
- 可开放一个公网 TCP 端口，例如 `3000`。
- 代码通过 git 或压缩包部署。
- 不依赖数据库。
- 房间状态默认只存在内存里。

### 12.2 启动命令

服务器上运行：

```bash
node poker.js server --host 0.0.0.0 --port 3000
```

推荐 P1 支持：

```bash
PORT=3000 HOST=0.0.0.0 node poker.js server
```

### 12.3 防火墙

服务器需要放行游戏端口：

- 云厂商安全组放行 TCP `3000`。
- 服务器本机防火墙放行 TCP `3000`。

服务端启动时，如果端口监听失败，需要给出明确错误：

- 端口已占用。
- 权限不足。
- 地址不可绑定。

### 12.4 进程守护

P0 可以手动启动服务端。

P1 建议支持 systemd 或 pm2：

```bash
pm2 start poker.js -- server --host 0.0.0.0 --port 3000
```

服务端意外退出时应由进程守护工具重启。由于 P0 不持久化房间，重启后原房间丢失是可接受的，但客户端必须显示「服务器已断开」。

### 12.5 日志

服务端至少输出：

- 启动时间。
- 监听地址和端口。
- 房间创建。
- 玩家连接和断开。
- 单手开始和结束。
- 非法消息和错误。

日志不能输出：

- 未公开手牌。
- 完整牌堆。
- 重连码明文，除非是启动或调试模式。

## 14. 安全与公平

- 洗牌、发牌、AI 决策和结算全部在服务端执行。
- 客户端永远不能收到他人未公开手牌。
- 客户端动作必须校验：
  - 是否轮到该座位。
  - 动作是否在合法动作列表中。
  - 金额是否符合下注、加注、全下规则。
  - `handNo` 和 `stage` 是否匹配当前状态。
- 同一玩家重复提交同一个 `clientActionId` 时，服务端应幂等处理。
- 房主权限只影响房间管理，不影响发牌和胜负。
- 房间码和重连码需要足够随机，避免被轻易猜中。
- 2.0 部署在公网服务器上，但只面向朋友局，不做公开大厅，不承诺反作弊到竞技平台级别。
- 服务端应限制单 IP 连接数，防止误操作或简单刷连接拖垮进程。
- 管理口令只允许创建房间和管理服务，不允许查看隐藏牌局信息。

## 15. 异常处理

- 玩家断线：保留座位并广播离线状态。
- 当前行动玩家断线或超时未行动：倒计时结束后默认弃牌，不自动过牌。
- 房主断线：房间继续存在；如果 5 分钟未重连，可转移房主给最早在线玩家。
- 服务端收到非法动作：拒绝并返回错误，不改变牌局状态。
- 客户端重连：优先用本地 sessionId，失败时允许输入重连码。
- 房间不存在：提示房间已关闭或不存在。
- 服务端关闭：客户端提示连接断开并退出。
- 网络消息格式错误：服务端忽略该消息并返回错误。
- 服务器重启：所有内存房间丢失，客户端提示房间已结束。
- 端口不可达：客户端提示检查服务器地址、端口和防火墙。

## 17. 测试重点

### 17.1 核心逻辑

- 原 1.0 单机测试继续通过。
- 多真人座位下下注顺序正确。
- 真人和 AI 混合时阶段推进正确。
- 边池结算正确。
- 水下模式正确。
- AI 行动仍不读取隐藏信息。

### 17.2 服务端

- `server` 能在 `0.0.0.0:3000` 启动。
- `create` 能连接服务器并创建房间。
- `join` 能连接并加入房间。
- 房间码校验正确。
- 加入不存在房间会失败。
- 房主权限校验正确。
- 非当前行动玩家不能行动。
- 非法下注金额不会改变状态。
- 重复动作请求不会重复下注。
- 同一房间多个客户端收到一致公共状态。
- 不同玩家收到不同私有手牌。
- 玩家断线不破坏牌局状态。
- 服务器重启后客户端能识别断线。
- 端口不可达时客户端错误清晰。

### 17.3 客户端

- 等待区座位状态实时刷新。
- 玩家行动提示只在轮到自己时出现。
- 输入非法命令时有明确错误。
- 断线重连后能恢复牌桌状态。
- 单手结束后复盘展示正确。
- 服务端关闭时客户端能正常退出。

### 17.4 部署

- 服务器开放端口后，外网终端可连接。
- 服务端在后台运行时，多个客户端可同时加入。
- 云安全组或防火墙未开放端口时，客户端能提示连接失败。
- 公司电脑不需要开公网监听，不需要运行 ngrok 或 Tailscale。

## 18. 验收标准

2.0 完成时，必须满足：

- 个人服务器可以用 `node poker.js server --host 0.0.0.0 --port 3000` 启动联机服务。
- 房主可以用 `node poker.js create your-server.com:3000` 创建命令行联机房间。
- 朋友可以用 `node poker.js join your-server.com:3000 房间码` 加入房间。
- 至少 2 名真人玩家可以在不同终端里完成一手牌。
- 房主可以用 AI 补足座位，并调整 AI 名称和难度。
- 真人和 AI 可以在同一桌正常完成多手牌。
- 服务端统一决定发牌、行动校验和结算。
- 玩家只能看到自己的暗牌，不能通过客户端状态看到他人暗牌或牌堆。
- 断线后在房间生命周期内可以回到原座位。
- 原有单机命令行模式没有回退。
- 公司电脑无需开放公网端口或运行公网隧道即可参与开发和测试。
